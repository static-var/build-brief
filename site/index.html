Gradle output, distilled.

build-brief sits in front of Gradle, keeps the full raw log on disk, and returns the signal that usually matters: failures, warnings, generated artifacts, and final build status.

Inspired by rtk, but built separately because RTK is not compatible enough with Gradle workflows to reuse directly.

How it works

Three steps. Nothing hidden.

01

Wraps Gradle

Resolves an explicit --gradle path, an explicit build-brief gradle ... or build-brief ./gradlew ... invocation, then falls back to ./gradlew and finally system gradle. It normalizes output-shaping flags to keep parsing stable.

02

Logs everything to disk

Streams the full raw output to a reusable per-project log file. Nothing is thrown away. Memory use stays steady regardless of output size.

03

Returns the signal

Default output shows failed tasks, failed test names, warnings, generated artifacts, assertion details, and build status.

Use `build-brief` for routine Gradle commands.

- Prefer `build-brief gradle ...` for a PATH-resolved Gradle binary.
- Prefer `build-brief ./gradlew ...` when the repo has a Gradle wrapper.
- Let `build-brief` keep `--console=plain` and normalize output-shaping flags.
- Do not add `--daemon` or `--no-daemon`; let normal daemon reuse happen.
- Fall back to raw Gradle only when the reduced summary is not enough.

Claude Code hook

FAQ

Common questions.

Does it replace Gradle?

No. It wraps Gradle, preserves the exit code, and keeps the raw output.

Does it hide stack traces or failures?

No. Failures stay visible, and the raw log path is printed when you need the full output.

How accurate are the savings numbers?

Approximate. It uses a chars-divided-by-4 estimate and is meant for rough savings/accounting feedback.

How does daemon reuse work?

build-brief now lets Gradle reuse an existing daemon or start one when needed. It strips explicit --daemon and --no-daemon overrides so reducer behavior stays consistent.

Why not just use RTK?

build-brief is inspired by RTK, but RTK is not compatible enough with Gradle workflows for this use case.

Where might it not work as expected yet?

The happy path is strongest for standard Gradle command shapes and output locations. Unusual plugins, custom artifact directories, or odd wrappers may still need the raw log as fallback.

Is it heavily tested on every platform?

Not yet. The project targets macOS, Linux, and Windows, but it has not been heavily tested across every OS, shell, CI environment, and Gradle/plugin combination so far. Some bugs and rough edges are still expected.

How do updates work?

If you installed with Homebrew, run brew upgrade build-brief. Otherwise, grab the newest build from the latest release.
