Keep the log. Lose the wall of text.

build-brief sits in front of Gradle, keeps the full raw log on disk, and returns the parts people and coding agents actually need: failures, warnings, test summaries, and final status.

why teams use it

Gradle output is fine until it becomes everyone’s problem.

It’s not just a readability thing. Long Gradle output slows down debugging, buries the actual failure, and burns tokens when coding agents keep re-running builds in a loop.

Readable by default

Successful runs stay short. Failing runs surface the failed tasks, test names, warnings, and the final result without making you scroll through pages of noise.

Agent-friendly on purpose

The tool is built for coding loops. That means less junk in terminal transcripts and less wasted context when agents keep hitting Gradle commands.

Raw logs stay available

Nothing important is thrown away. `build-brief` streams the full log to disk and points you to it whenever the short summary is not enough.

Works across Gradle stacks

Spring Boot, Ktor, Android, Kotlin Multiplatform, plain JVM, and multi-module projects all work because the tool wraps the Gradle process itself.

especially useful when

You are running build loops all day

If you bounce between `test`, `build`, `assemble`, and failing retries, the gain is immediate. Less scroll. Faster failure recognition. Cleaner terminal history.

especially useful when

You are using AI tools against a Gradle repo

Coding agents do not need every task line and daemon blip. They need signal. `build-brief` was built to preserve that signal while keeping the raw log one step away.

faq

The usual questions.

Does it replace Gradle?

No. It wraps Gradle, keeps the exit code, and preserves the raw output. The goal is a better default terminal view, not a new build system.

Does it hide stack traces or real failures?

No. Failures stay visible. If the short summary is not enough, the raw log path is already there so you can inspect the full output.

Is this only for one type of Gradle project?

No. It works at the Gradle process and log level, so it is designed for Spring Boot, Ktor, Android, KMP, plain JVM projects, and multi-project builds.

Why use this with coding agents?

Because agents tend to re-run noisy commands over and over. Short summaries make it easier for them and for you to spot what changed without flooding the transcript.

Which domain should we use?

I’d use bb.staticvar.dev. It is shorter, easier to say out loud, and still maps cleanly to “build brief.” If you prefer the more explicit version, `gbb.staticvar.dev` is still easy to support later.
